Engineering a 100/100 Portfolio

April 25, 2026·8 min read·1,042 words

In the modern web, performance is often treated as a secondary metric to be "fixed" after launch. For this portfolio, performance was the foundational constraint. The goal: a "Full House" Lighthouse score (100/100 across the board) while maintaining high-fidelity visuals like a GPU-accelerated starfield and 3D hero interactions.

To achieve this, I had to abandon the comforts of modern frameworks and return to the metal: vanilla HTML, CSS, and a surgical implementation of JavaScript. This post breaks down the core pillars of that architecture.

1. Main Thread Shielding

The biggest threat to a 100 Performance score is TBT. Heavy animations, even when GPU-accelerated, can jam the main thread during the critical audit window. We solved this by implementing a shielding strategy using requestIdleCallback.

Most developers initialize their "heavy" logic on DOMContentLoaded. While this feels fast, it spikes the CPU right when Lighthouse is trying to measure interactivity. Instead, I moved the Starfield and Hero engine initialization into a background-priority queue:

// Deferring non-critical visuals
window.addEventListener("load", () => {
  const init = () => {
    renderStars(); // High-count particle system
    startHeroEngine(); // 3D transform listener
  };

  // requestIdleCallback
  if (window.requestIdleCallback) {
    requestIdleCallback(init, { timeout: 2000 });
  } else {
    setTimeout(init, 200);
  }
});

By delaying the initialization until the browser is idle, we reduced TBT to exactly 0ms. This ensures the FCP and LCP metrics are recorded before the particle system begins its heavy math cycles.

2. Enterprise-Grade Security Hardening

Security headers aren't just for compliance; they are a statement of technical maturity. We implemented a strict Content Security Policy (CSP) and optimized for HSTS Preloading. Many sites fail their "Best Practices" audit because of insecure cross-origin connections or lack of strict transport protocols.

I configured my Netlify _headers file to enforce a "Zero-Trust" environment. By setting a 2-year max-age with the preload flag, this domain is hard-coded into browsers as HTTPS-only:

/*
  Strict-Transport-Security: max-age=63072000; includeSubDomains; preload
  X-Frame-Options: DENY
  X-Content-Type-Options: nosniff
  Referrer-Policy: strict-origin-when-cross-origin
  Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline' fonts.googleapis.com; font-src 'self' fonts.gstatic.com; img-src 'self' data:;

This configuration eliminates the latency of initial HTTP-to-HTTPS redirects and mitigates Man-in-the-Middle (MITM) attacks before they can even initiate a handshake.

3. Zero-JS Micro-Interactions

To keep the bundle size near zero, I moved UI depth into the CSS layer. Every interaction on the site, from the "Traits" tooltips to the glassmorphism modals, is powered by pure CSS logic. This isn't just about speed; it's about durability. CSS-based interactions work even if the user has a spotty connection and the JS fails to load.

The tooltips use ::after pseudo-elements and the content: attr(data-tooltip) property. This allows for rich, glassmorphism-styled hover effects with zero impact on the JavaScript execution budget:

[data-tooltip]::after {
  content: attr(data-tooltip);
  backdrop-filter: blur(12px);
  background: rgba(15, 15, 15, 0.8);
  /* The rest is positioning and opacity math */
}

By using backdrop-filter sparingly and only on hover, we maintain a premium feel without triggering constant "layout shifts" that would hurt our CLS score.

4. Semantic Discovery for AI Agents

As we move toward an agentic web, technical SEO must evolve. Traditional SEO focuses on keywords for humans; Agentic SEO focuses on context for LLMs. We implemented JSON-LD Person Schema to explicitly link this domain to my professional entities (GitHub, LinkedIn).

Furthermore, I provided a dedicated /llms.txt file. This is a new standard designed specifically for AI crawlers like GPT-5.5 or Claude Opus 4.7. It provides a high-density, markdown-based summary of my technical stack, projects, and contact info, ensuring that when an AI is asked about "Ocean Bennett," it provides an accurate and up-to-date response.

5. Offline-First PWA Strategy

A portfolio should be a reliable tool, not just a document. By implementing a custom Service Worker, I've enabled full offline support. The site uses a "Stale-While-Revalidate" strategy for assets and a "Cache-First" strategy for the core shell.

This means that once a user visits the site, the entire blog engine, including the posts.json metadata and the article template, is saved locally. Even if the user enters an area with no service, the search and writing sections remain fully functional. This creates an "App-like" feel that is rare in personal portfolios.

TL;DR: The Essentials

achieved 100/100 Lighthouse scores by shielding the main thread with requestIdleCallback, hardening security with strict HSTS/CSP headers, and offloading all UI interactivity to Zero-JS CSS pseudo-elements. The site is optimized for AI agents via JSON-LD and llms.txt, and works entirely offline thanks to a custom Service Worker.
